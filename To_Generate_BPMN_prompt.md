Выступай в роли эксперта по моделированию процессов в нотации BPMN 2.0.

ЦЕЛЬ
- Из ИСТОЧНИКОВ выдели ОДИН бизнес-процесс и создай диаграмму BPMN 2.0 в формате XML, которую можно открыть в стандартных BPMN 2.0 viewers (например, Camunda Modeler, bpmn.io).
- Результатом должен быть ОДИН валидный BPMN 2.0 XML document, представляющий процессный поток.

ИСТОЧНИКИ 
```поместите сюда текст или таблицу, или напишите, из каких файлов взять описание процесса```

ПРАВИЛА ИНТЕРПРЕТАЦИИ
1. Определи, какой процесс моделировать:
   - Если ИСТОЧНИКАХ имеется неоднозначность, выбери наиболее полно описанный end-to-end поток, подразумеваемый источниками и прикреплёнными файлами.

2. Определи для этого процесса:
   - Начальные и конечные условия → BPMN start / end events.
   - Ключевые активности (действия пользователя, вызовы сервисов, переходы состояний, фоновые задачи) → BPMN tasks.
   - Точки принятия решений / логика ветвления → BPMN gateways.
   - Любые очевидные повторно используемые подпроцессы → subprocesses или call activities (опционально, только если это явно обосновано).

3. Уровень детализации:
   - Каждая задача (шаг процесса) должна быть значимой единицей работы.
   - Control flow (ветвления, слияния, циклы) должен быть явным и читаемым.
   - Используй метки, понятные бизнесу, вместо технических имён.

СТРУКТУРА BPMN 2.0 XML
Создай BPMN 2.0 XML document со следующей структурой:

- Заголовок XML:
  - `<?xml version="1.0" encoding="UTF-8"?>`

- Корневой элемент:
  - `<bpmn:definitions>` как минимум с этими атрибутами:
    - `xmlns:bpmn="http://www.omg.org/spec/BPMN/20100524/MODEL"`
    - `xmlns:bpmndi="http://www.omg.org/spec/BPMN/20100524/DI"`
    - `xmlns:di="http://www.omg.org/spec/DD/20100524/DI"`
    - `xmlns:dc="http://www.omg.org/spec/DD/20100524/DC"`
    - `xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"`
    - `id` (например, `Definitions_1`)
    - `targetNamespace` (например, `http://bpmn.io/schema/bpmn`)

- Элемент процесса:
  - Один дочерний элемент `<bpmn:process>` со следующими параметрами:
    - `id`, например `Process_Main` или идентификатор на основе области, такой как `Process_Checkout`.
    - `name`, описывающий процесс, желательно полученный ИСТОЧНИКОВ.
    - `isExecutable="false"` по умолчанию, если контекст явно не указывает на executable process для workflow engine, в этом случае допустимо `true`.
  - Внутри процесса:
    - Ровно один start event (например, `<bpmn:startEvent id="StartEvent_1" name="...">`).
    - Один или несколько end events.
    - Последовательность tasks (`bpmn:task`, `bpmn:userTask`, `bpmn:serviceTask` и т.д.).
    - Gateways (`bpmn:exclusiveGateway`, `bpmn:parallelGateway` и т.д.) для ветвления или слияния, где требуется.
    - Sequence flows (`bpmn:sequenceFlow`), соединяющие все элементы от start event до end event(s), с валидными `sourceRef` и `targetRef`.

- Диаграмма / разметка (BPMNDI):
  - Элемент `<bpmndi:BPMNDiagram>`, содержащий:
    - `<bpmndi:BPMNPlane bpmnElement="<process-id>">`, ссылающийся на id процесса.
    - `<bpmndi:BPMNShape>` для каждого визуального узла (start/end events, tasks, gateways), каждый со следующими параметрами:
      - `id` (например, `_BPMNShape_StartEvent_1`) и `bpmnElement`, указывающий на соответствующий id элемента BPMN.
      - Один `<dc:Bounds x="..." y="..." width="..." height="..."/>` для его позиционирования.
    - `<bpmndi:BPMNEdge>` для каждого sequence flow, каждый со следующими параметрами:
      - `id` (например, `<sequence-flow-id>_di`) и `bpmnElement`, ссылающийся на id потока.
      - Один или несколько `<di:waypoint xsi:type="dc:Point" x="..." y="..."/>`.

СОГЛАШЕНИЯ ПО МОДЕЛИРОВАНИЮ
- Аннотация области:
  - В самом верху XML, сразу после заголовка, включи XML-комментарий вида:
    - `<!-- BPMN модель для: <scope> -->`
  - В противном случае сформулируйте краткую строку scope на основе понимания процесса.

- Именование:
  - Process id: `Process_<CamelCaseOrSnakeCaseScope>`, где возможно.
  - Task ids: `Task_<ShortLabel>` (только буквы, цифры и подчёркивания).
  - Gateway ids: `Gateway_<ShortLabel>`.
  - Start / end event ids: `StartEvent_1`, `EndEvent_1` и т.д.
  - Sequence flow ids: `SequenceFlow_1`, `SequenceFlow_2`, ...

- Типы задач:
  - Используйте `bpmn:userTask` для явных шагов с участием человека, если это явно действия пользователя.
  - Используйте `bpmn:serviceTask` для отдельных вызовов сервисов/API, фоновых задач или автоматизированной работы системы.
  - Используйте общий `bpmn:task`, когда тип не очевиден или смешивание аспектов ввело бы в заблуждение.

- Шлюзы:
  - Используйте `bpmn:exclusiveGateway` для решений типа "или/или" (если X, то A, иначе B).
  - Используйте `bpmn:parallelGateway`, когда несколько путей выполняются параллельно и позже соединяются.

ЭВРИСТИКИ РАЗМЕЩЕНИЯ (ДЛЯ BPMNDI, ВАЖНО!)
- Размести основной happy path слева направо по горизонтали:
  - Start event около `x ≈ 150, y ≈ 120`.
  - Последовательные tasks с интервалом ~150 единиц по оси x (при том же y).
  - End event около правой стороны диаграммы.
- Для ветвлений:
  - Размести gateway на основном пути.
  - Размести альтернативные ветви выше и/или ниже с разными координатами y.
- Убедитесь, что каждый элемент BPMN, появляющийся в `<bpmn:process>`, имеет соответствующую shape или edge в `<bpmndi:BPMNPlane>`, чтобы стандартные BPMN 2.0 viewers могли отобразить диаграмму без ошибок.

КОНТРОЛЬНЫЙ СПИСОК ПРОВЕРКИ ПЕРЕД ОТВЕТОМ
Перед ответом внутренне проверьте, что:
- XML хорошо сформирован: теги правильно вложены и закрыты.
- Все ссылочные ids существуют:
  - Каждый `sourceRef` / `targetRef` ссылается на существующий элемент BPMN.
  - Каждый атрибут `bpmnElement` в `bpmndi:BPMNShape` / `bpmndi:BPMNEdge` соответствует существующему id элемента BPMN.
- Имеется ровно одно start event и как минимум одно end event.
- Нет отключённых tasks или gateways.
- Нет текста-заглушки типа "TODO" или "TBD".

ФОРМАТ ВЫВОДА
- в ответ включи только XML; не пиши комментариев и предложений по дальнейшей работе.

Теперь, используя приведённые выше правила и доступный контекст, сгенерируй BPMN 2.0 XML diagram.
